1 # Yarn Spinner Road Map
2 
3 This is the current road map we are considering for the future of Yarn Spinner.
4 Nothing in here is fixed and reflects what we believe should be the focus of development.
5 The idea of this document is to both give us something to help plan our development but to also give you all a space to see what we are doing, and to give us input on what you consider important.
6 
7 Let us know what you think about this road map either in the [Slack](http://lab.to/narrativegamedev) or [here on the GitHub issue we created about the road map](https://github.com/YarnSpinnerTool/YarnSpinner/issues/183).

11 # Releases
12 
13 Here are the main releases we're currently planning.
14 
15 ## V1.0
16 
17 ### Goal
18 
19 To make it easier for projects to spin up and remain compatible with core Yarn Spinner, and to smooth out the issues of getting started using and updating Yarn Spinner.
20 
21 ### Motivation
22 
23 Yarn Spinner has been in a lot of flux, more or less starting as a spin-off of Night In The Woods. A lot of it was heavily tied to the needs and deadlines of that specific game.
24 We've been working to make it more generic for _any_ narrative game, but there is still a lot of connections to the requirements of Night In The Woods.
25 As such, there is a lot of wonky code and half-finished concepts which users currently need to deal with. We want that to change so you no longer have to deal with this when it comes time to update or get started with Yarn Spinner.
26 
27 ### Features
28 
29 - [x] Formal Yarn specification created and frozen
30 - [x] Bytecode format freeze
31 - [x] Compiler will generate bytecode by default instead of parsing the dialogue each time
32 - [x] Default to using `.yarn` as the file extension in preference to `.yarn.text`
33 - [x] VM will load and execute compiled bytecode
34 - [x] Unity importer for `.yarn` files that invokes compiler and generates bytecode assets (using a `.yarnc` file extension).
35 - [x] Implementation of a barebones Unity prefab (with a Canvas) that can be added into a scene to "just work"
36 - [x] Rework examples to be designed to be used as the basis of your own game
37 - [x] Modify example dialogue runner and variable storage to be the base form, instead of "just" examples
38 - [ ] Syntax highlighter extension for Visual Studio Code
39 - [ ] New documentation site

41 ## V1.1
42 
43 ### Goal
44 
45 To add in the two most requested features: string interpolation, and localisation! String interpolation means being able to write lines something like `Hello, ${playerName}`, and have the line appear as "Hello, Sally"; localisation means being able to translate your game's content into multiple languages, keeping in mind the different rules that languages have regarding things like how words are pluralised, how grammatical gender works, and more.
46 
47 ### Motivation
48 
49 By far the most requested feature, string interpolation has been put off due to concerns about implementation, in a nutshell to do "proper" interpolation is far from trivial. We're also very aware of the requirements of people who are releasing their games on storefronts that require localisation.
50 
51 This release will have no other features beyond those necessary to support interpolation and localisation features, as we want to ensure this is reliable. We are also concerned as to how many bugs this may cause in existing games, so we're moving carefully here.
52 
53 ### Features
54 
55 - String interpolation
56 - Localisation support
57 - Updated examples
58 - Updated documentation

60 ## V1.2
61 
62 ### Goal
63 
64 To add in initial support for Unreal.
65 
66 ### Motivation
67 
68 Unreal is a large portion of the games community, and we've been asked multiple times by different people if Unreal is supported. We want to be able to say yes to them.
69 This is not to say it will be as featureful as the Unity version for V1.2, although the long term goal is for feature parity.
70 
71 ### Features
72 
73 - Create an Unreal VM capable of reading compiled Yarn bytecode
74 - Create Unreal Dialouge Runners and Variable Storage
75 - Create examples showing how to use Yarn Spinner in Unreal.
76 - Add documentation on using Yarn Spinner in Unreal.
77 
78 ## V1.3
79 
80 ### Goal
81 
82 To create resources for developers to get more out of using Yarn Spinner.
83 
84 ### Motivation
85 
86 At this point, Yarn Spinner should be at a state where it is stable to use and easy to update.
87 The intent is to then make it more convenient for developers to integrate, so more examples and ready to go elements.
88 
89 ### Features
90 
91 - Modular prefab UI elements designed to work out of the both with Yarn Spinner
92 - More examples and example projects demonstrating using Yarn in different forms
93 - Add in parsing `.yarn` files into Unreal
94 
95 ## V1.4
96 
97 ### Goal
98 
99 Make Yarn more accessible from your existing development tools.
100 
101 ### Motivation
102 
103 While we expect the [Yarn editor](https://github.com/YarnSpinnerTool/YarnEditor) will remain the main way you will create and modify Yarn files, there are always more workflows that need support.
104 This release will be designed to try and help make that easier.
105 
106 ### Features
107 
108 - Merge all command line tool functionality into the Unity and Unreal core
109 - Create a [language server protocol](https://microsoft.github.io/language-server-protocol/) server for Yarn
110 - Add diagnostics and debugging features to a VSCode extension
111 - Create a WebAssembly compiler for Yarn, allowing browsers to run the exact same code to compile Yarn as is used in Unity and Unreal
112 
113 ## V1.5
114 
115 ### Goal
116 
117 To put Yarn Spinner in a place where it will be capable of being used more easily and in multiple places.
118 
119 ### Motivation
120 
121 We want Yarn Spinner to be easy to implement, use, and port to as many places as possible.

146 *Why are there no dates attached to the releases?*
147 
148 Yarn Spinner is a community project and we (as in @desplesda and @McJones) as the main devs work on it in our free time or when it aligns with work.
149 Any Yarn Spinner work has to fit around our own work and as such would be the first candidates to be dropped if we need more time for work or our own sanity.
150 We don't want to put down deadlines that, at least at this stage, would be known to be arbitrary.
151 
152 *I think Feature X is the most important and you've not got it listed, what gives?*
153 
154 This is what we think is important, but we want to know what **you** think is important, so either let us know in the Slack or on the [issue we opened about the road map](https://github.com/YarnSpinnerTool/YarnSpinner/issues/183).
155 We want to build Yarn Spinner into something helpful for you as well as us.
156 
157 *I think Feature Y is the most important but you've got it way down the bottom of the road map, what gives*
158 
159 We've tried to group the road map releases into features and changes in a manner that we think makes sense, but we want your feedback on this stuff, so let us know.
160 
161 *You've mentioned this Slack now a few times, how do I get in?*
162 
163 You can join the Narrative Game Dev slack [here](http://lab.to/narrativegamedev). We hope to see you there!
164 
165 *What about bug fixes? I still have a bug you've not fixed yet!*
166 
167 Throughout all of the releases the intent is to try and keep the bugs to a minimum.
168 Some of these changes are intentionally made to clean up some of the cruft we've accumulated that make bug fixes hard.
169 
170 *Why does the road map get less detailed the longer it goes on?*
171 
172 Some of these changes we can see exactly how they will work and integrate, others will need to change as we get closer.
173 Later releases are designed to have much larger broad strokes of features instead of specifics like in V1.0.
174 As a release comes out we can take a look at the road map again, work out what features and needs should go into the next release and update the road map appropriately.
